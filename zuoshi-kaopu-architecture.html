Section B: 搜证 独立可调

搜证
gather sources
找材料

WebFetch 返回 AI 摘要，实测丢失 93% 内容。后面 NLM 质证需要原文，所以搜集阶段必须拿到忠实于原文的版本。分两阶段：宽口快筛 + 窄口深度抓取，平衡效率和保真度。

1
搜索
WebSearch 按研究方向搜索，拿到 URL 列表
发现候选源
技术: WebSearch
2
快筛
WebFetch 对每个 URL 做快速摘要，判断相关性，筛选高价值源
AI 摘要在判断"要不要深入"时是对的工具，200 字够判断相关性。不需要对每个 URL 都深度抓取
技术: WebFetch (AI 摘要, 有损但此处够用)
?
筛选数量动态调节
不是固定数字。取决于 Coverage Matrix 空格数、信息饱和度、研究深度
3
深度抓取
对筛选后的高价值 URL，三层瀑布流抓取完整原文
WebFetch 丢 93% 内容。curl + textutil 或 Playwright 拿到的是完整正文，没有 AI 处理
Layer 1: curl + textutil
大部分站点可用。macOS 原生，零依赖。拿到原始 HTML 转纯文本。
Layer 2: Playwright headless
curl 拿不到时（403 / JS 渲染）。headless Chromium 渲染后提取 article 正文，更干净。
Layer 3: 记录到 unfetched
Playwright 也过不去（WAF / 验证码 / 登录墙）。记录到 raw/00-unfetched-sources.md，标记手动获取。
4
本地落盘
抓取结果存到 raw/extracted/，文件头标注来源 URL + 抓取日期 + 抓取方式
本地副本是一切分析的地基，也是 NLM URL 灌不进去时的兜底材料
5
灌入 NLM
NLM source_add(url) 让 NLM 自己抓取原文
NLM 的原文副本是后续质证的权威源
技术: mcp__notebooklm-mcp__source_add
?
NLM 加载是否成功？
成功
正常流程。质证用 NLM 原文。
失败（中文站常见）
source_add(file, raw/extracted/xxx.md) 上传本地副本作为替代源。NLM 支持 MD/TXT/PDF 上传。
产出物
raw/extracted/*.md 本地原文副本（带元数据头）
raw/00-unfetched-sources.md 无法自动获取的源列表
NLM notebook 中已加载的源（URL 或文件）

Section C: 脑暴 独立可调

脑暴
brainstorm
challenge this

用第二个模型做对抗性挑战。第二模型没参与你的推理过程，看不到你的确认偏误，所以能找到你自己找不到的盲区。关键约束：只许提问和找漏洞，不许给替代方案。

1
理解上下文
用户带着具体内容说"脑暴"，直接用。只说"脑暴"，问一句"你在想什么？哪个地方不踏实？"
接受半成型的想法，不要求正式的 research question
2
发送挑战
把用户的思考发给第二模型，附对抗性指令：找盲区、拆假设、提未考虑的问题。每个发现分类为逻辑风险 / 证据风险 / 执行风险
分类帮助用户判断优先级：证据风险可以质证解决，逻辑风险需要重新思考
技术: Codex CLI / 第二模型 API / 跨 agent 调用
3
返回发现
呈现挑战结果。提示"这里面有几个论断可能值得质证，要继续吗？"
自然过渡到质证环节，但不强制
产出物
挑战发现列表（每条含：被挑战的点 + 为什么是问题 + 风险类型）
在完整研究流程中写入 04-challenge.md
对抗协议六条：
  1. 找盲区和未声明的假设
  2. 找逻辑缺口和弱推理
  3. 提用户没考虑过的硬问题
  4. 每条分类：逻辑风险 / 证据风险 / 执行风险
  5. 不许给替代方案
  6. 不许表扬。你的工作是找问题

Section D: 质证 独立可调

质证
verify
check against sources

LLM 是假设引擎，NLM 是证据引擎。LLM 提假设，NLM 用原文回答。反过来不行。NLM 自己的综合性回答不是证据，只有它引用的原文才是。

1
确定待验证声明
用户指定具体声明，或从脑暴结果中提取证据风险项
有针对性地验证，不做泛泛检索
2
确认源已加载
检查 NLM notebook 中是否有相关源。缺的补上（URL / 文件 / 文本）
没有源就没有证据
技术: source_add(url | file | text)
3
查询验证
把声明转化成 NLM query。提取阶段和验证阶段分开：先查一遍收集候选证据，再换问法重新查确认
同一个问题换问法能暴露第一遍漏掉的证据或矛盾
技术: mcp__notebooklm-mcp__notebook_query
4
三层检查
对每条证据做三层独立验证：链接是否有效 → 内容是否相关 → 事实是否支撑声明
研究表明链接有效率 94%+，但事实支撑率只有 39-77%。"链接在"不等于"内容支撑"。表面指标会骗人
5
出具证据卡
每条声明生成一张证据卡：声明 / 判定(supported, not found, contradicted) / 原文引用 / 来源 / 置信度 / 此证据不能证明什么
最后一栏"不能证明什么"防止过度解读
产出物
Claim Ledger（每条含 ID / 声明 / 状态 / 原文引用 / 来源 / 置信度）
在完整研究流程中写入 05-evidence.md
信源分级：
A一手官方文件年报、法院文件、SEC 备案
B官方网站 / 自述公司官网、LinkedIn、个人声明
C第三方数据库Bloomberg、Crunchbase
D新闻 / 二手报道媒体文章、Wikipedia
X推断 / 假设禁止使用。标记"无公开来源"

研究

说"研究 + 话题"时触发。按 methodology.md 工序顺序调用 Section B/C/D，加上契约、综合、自检环节。不是单独的 Section，是对三个模块的编排。

契约

决定模式：Lightweight / Standard / Audit

脑暴 + 质证（并行）

技术栈

能力工具用在哪
网页搜索 WebSearch 搜证 Step 1：发现候选 URL
快速筛选 WebFetch 搜证 Step 2：AI 摘要判断相关性（有损，此处够用）
原文抓取 L1 curl + textutil 搜证 Step 3：静态站原文。macOS 原生，零依赖
原文抓取 L2 Playwright (Chromium) 搜证 Step 3：JS 渲染站 / curl 403 兜底
证据引擎 NotebookLM via MCP 搜证 Step 5 灌源 / 质证全流程 / 源文引用验证
NLM 工具集 mcp__notebooklm-mcp__* source_add / notebook_query / notebook_create
对抗性挑战 Codex CLI / 第二模型 API 脑暴：跨模型盲区发现
自我对抗（备用） Sub-agent / 自我角色切换 脑暴：无第二模型时的降级方案
过程文档 Markdown 文件链 01-brief → 02-coverage → 03-hypotheses → ... → 07-deliverable

三种模式

模式触发条件差异
Lightweight 理解 + 快速概览 Coverage Matrix 内联，跳过脑暴，3问自检，单文件产出
Standard 理解 + 独立判断 完整 Coverage Matrix，完整脑暴+质证，5问自检，4文件产出
Audit 为了做决定 完整文件链 00-07，Claim Ledger 贯穿全程，Standard Changelog 可选
